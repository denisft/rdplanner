# БЛ-4 «Сдвиг начала горизонта» — план реализации

Детальная архитектура фичи: автопрокрутка ганта к «сегодня» + кнопка «Сегодня»,
кнопка «К текущей неделе» (сдвиг-снимок начала горизонта), плашка-напоминание
и правило движка «прошлое прожито».

Ссылки: постановка — `resource-planner-BACKLOG.md`, п. БЛ-4 (требования D.1–D.8).
Смежное — БЛ-3 (план/факт), БЛ-11 (undo для всех правок).

## Ключевой принцип

**Сдвиг начала горизонта = фиксация прошлого, не перепланирование.** Тот же
приём, что в `completeTask` (`src/engine/complete.ts`): берём текущий
`ScheduleResult`, закрепляем на рассчитанных датах всё, что начинается раньше
новой границы, и только потом меняем `horizonStart`. Движок дополняется
правилом: закреплённый этап, начавшийся до начала горизонта, не
переразмещается — прожитая часть отрезается, хвост продолжает занимать
исполнителя на прежних датах.

Главный инвариант (требование D.4, фиксируется тестом):

```
release(schedule(data)) === release(schedule(shiftHorizonStart(data, result, newStart)))
```

— ни одна дата релиза активной задачи не меняется. Сдвиг лишь отрезает
прошлое: завершённые задачи, целиком уехавшие за границу, исчезают с ганта
(правило «прошлое не воскресает» уже есть в `scheduler.ts:156`), гант очищается.

Модель `AppData` не меняется — используем существующее поле
`Stage.pinnedStartDate`. `SCHEMA_VERSION` прежняя, старые файлы и
шаринг-ссылки совместимы без миграции (D.8).

## 1. Даты: `src/engine/dates.ts`

```ts
/** Число рабочих дней в полуинтервале [fromISO, toISO) —
    насколько «в прошлое» от начала горизонта уехал старт этапа. */
export function workingDaysBefore(fromISO: string, toISO: string): number {
  let count = 0;
  let cursor = parseISO(fromISO);
  const end = parseISO(toISO);
  while (cursor < end) {
    if (!isWeekend(cursor)) count++;
    cursor = addDays(cursor, 1);
  }
  return count;
}
```

`addDays` сейчас приватная — экспортировать (нужна и в `horizon.ts` для
`shiftTarget`).

## 2. Движок: правило «прошлое прожито» в `src/engine/scheduler.ts`

### 2.1. `ScheduledStage` — новое производное поле

```ts
/** true — этап начат до начала горизонта: рисуется от левого края
    с обрезкой, реальный старт — в stage.pinnedStartDate. */
clippedStart?: boolean;
```

Поле производное (в `AppData` не попадает). Соглашение о `startIndex` не
ломаем: `-1` по-прежнему значит «не в сетке», обрезанный этап получает
`startIndex = 0` + флаг, отрицательных индексов не заводим.

### 2.2. Проход 1 (резервирование закреплённых): обрезка

Сейчас `dayToIndex.get(pinnedStartDate) === undefined` → `continue` (пин
«проваливается» в авторежим — то самое поведение, которое отвергнуто).
Новая логика:

```ts
let start = dayToIndex.get(stage.pinnedStartDate);
let clipped = false;
if (start === undefined) {
  // Дата вне сетки. Будущее за 260 дней / выходной — как раньше, авторежим.
  if (stage.pinnedStartDate >= days[0]) continue;
  // Прошлое: прожитую часть отрезаем, хвост занимает исполнителя (D.5).
  const lived = workingDaysBefore(stage.pinnedStartDate, days[0]);
  if (stage.durationDays - lived <= 0) continue; // целиком в прошлом → проход 2
  start = -lived; // виртуальный старт до сетки
  clipped = true;
}
const end = start + stage.durationDays - 1;
pinned.set(stage.id, { start: Math.max(0, start), end, clipped });
// занятость — только видимая часть:
for (let d = Math.max(0, start); d <= end && d < maxIndex; d++) { busy.add(d); markOccupied(...); }
```

В `pinned`-map добавляется флаг `clipped`, наружу в проходе 2 он уходит в
`ScheduledStage.clippedStart`.

Наложение обрезанного хвоста с другим закреплённым этапом — обычная
перегрузка через `occupancy`, отдельного кода не требует.

### 2.3. Проход 2: «целиком в прошлом» — теперь для всех задач

Существующая ветка «прошлое не воскресает» (`scheduler.ts:156`) сейчас
работает только для `done`. Обобщаем: закреплённый этап, **целиком**
оставшийся в прошлом (не попал в `pinned`-map), выкидывается из расчёта у
любой задачи:

```ts
if (
  stage.pinnedStartDate &&
  !pinned.has(stage.id) &&
  stage.pinnedStartDate < data.horizonStart &&
  (done || stage.assigneeId)   // у активных — только этапы с исполнителем (см. ниже)
) { /* emit со startIndex = -1, pinned: true; continue */ }
```

Важно: `prevEnd` при этом не трогаем — следующий этап задачи может начинаться
с начала горизонта (`earliest = prevEnd + 1`, а `findSlot` уже клампит к 0).
Warning'и по таким этапам не выдаём — это прошлое, а не проблема.

Этап без исполнителя с пином в прошлом у активной задачи — оставляем текущее
поведение (пин без исполнителя мёртв, этап кладётся авторежимом после
предыдущего): `completeTask` такие этапы и не закрепляет, а `shiftHorizonStart`
(§3) — тоже, так что в новую ветку они попадать не должны; условие просто
страхует от руками испорченных данных.

### 2.4. Что это меняет для завершённых задач

Раньше этап `done`-задачи, **пересекающий** границу, выбрасывался целиком
(исполнитель освобождался — неточность). Теперь его хвост занимает
исполнителя, как у активных. Существующий тест «прошлое не воскресает»
(`scheduler.test.ts:317`, пин 2026-05-18 при горизонте с 2026-06-01) остаётся
зелёным, если этап целиком в прошлом — проверить длительность в фикстуре;
если он пересекает границу, тест обновить осознанно (новое поведение
правильнее).

## 3. Сдвиг-снимок: `src/engine/horizon.ts` (новый)

Чистый модуль без React, по образцу `complete.ts`.

```ts
import type { AppData } from '../types';
import type { ScheduleResult } from './scheduler';
import { addDays, formatISO, parseISO } from './dates';

/** Целевая дата сдвига: понедельник ПРЕДЫДУЩЕЙ недели — одна неделя
    прошлого остаётся видимой для контекста (D.3). */
export function shiftTarget(todayISO: string): string {
  const d = parseISO(todayISO);
  const dow = (d.getDay() + 6) % 7; // пн = 0
  return formatISO(addDays(d, -dow - 7));
}

/** Сдвиг начала горизонта с фиксацией прошлого: всё, что по текущему
    расчёту начинается раньше newStart, закрепляется на своих датах
    (по образцу completeTask), затем переносится horizonStart.
    Инвариант: даты релизов не меняются. */
export function shiftHorizonStart(
  data: AppData,
  result: ScheduleResult,
  newStart: string,
): AppData {
  const startByStage = new Map<string, string>();
  for (const s of result.scheduledStages) {
    if (s.startIndex >= 0 && !s.clippedStart) {
      startByStage.set(s.stageId, result.days[s.startIndex]);
    }
  }
  return {
    ...data,
    horizonStart: newStart,
    tasks: data.tasks.map((t) => ({
      ...t,
      stages: t.stages.map((s) => {
        // Уже закреплённые не трогаем (в т.ч. обрезанные — их реальный
        // старт и так в pinnedStartDate); без исполнителя пин мёртв.
        if (s.pinnedStartDate || !s.assigneeId) return s;
        const start = startByStage.get(s.id);
        return start && start < newStart ? { ...s, pinnedStartDate: start } : s;
      }),
    })),
  };
}
```

### Почему релизы не едут

Всё расписание слева от границы закрепляется на прежних местах → занятость
исполнителей идентична прежней. Авторазмещение справа от границы
детерминировано (жадный проход по приоритету) и при той же занятости
воспроизводит те же даты. Обрезка (§2) гарантирует, что закреплённое в
прошлом не переразмещается и не исчезает, пока у него есть хвост в горизонте.

Единственное допустимое отличие: сетка в 260 рабочих дней теперь начинается
позже, т.е. тянется дальше вправо — задача, которая раньше «не влезала в
расчётный горизонт» (warning, `startIndex = -1`), может влезть и получить
дату релиза. Это появление даты, а не сдвиг; инвариант-тест сверяет только
задачи, имевшие дату до сдвига.

## Крайние случаи и решения

- **Сдвиг назад / в будущее.** Не поддерживаем: кнопка активна только при
  `horizonStart < target`. Расширение прошлого — отдельная потребность, не
  заявлена.
- **Все этапы активной задачи в прошлом.** Задача исчезает с ганта, но дата
  релиза в таблице сохраняется: у целиком прожитых этапов конец считается от
  закреплённых дат напрямую (`addWorkingDays`, мимо сетки) — инвариант D.4
  держится в полном объёме. То же верно для завершённых задач в прошлом
  (раньше их релиз в таблице становился «—»). Такой задаче место в
  «Завершённых» — тимлид сам решает, завершить или вернуть в работу.
- **Пин на выходном дне** — невозможен через UI (сетка — рабочие дни);
  сравнение `pinnedStartDate < horizonStart` строковое и корректно, т.к. обе
  даты ISO.
- **`horizonStart` на выходном** — `generateWorkingDays` уже нормализует к
  следующему рабочему дню; `shiftTarget` всегда возвращает понедельник.
- **Повторные сдвиги.** Идемпотентно по построению: второй сдвиг на ту же
  дату ничего не закрепляет (всё слева уже закреплено) и не меняет
  `horizonStart`; кнопка при этом уже `disabled`.
- **Undo сдвига.** Один снимок `data` в стеке — откатывает и `horizonStart`,
  и все новые пины разом.
- **Шаринг.** `horizonStart` — часть `AppData`, зритель по ссылке видит план
  от той же даты, что владелец; автоскролл к «сегодня» работает и у него.
- **260 рабочих дней.** Сетка строится от `horizonStart` — без сдвига сегодня
  через год вышло бы за расчётный горизонт; регулярный сдвиг решает и это.
