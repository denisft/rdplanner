# ADR-001: Поддержка нескольких команд в одном планировщике

**Статус:** Принято · **Дата:** 2026-07-18 · **Решает:** Денис

## Контекст

Появилась вторая команда разработки. Нужно вести отдельный гант на каждую.
До этого `AppData` был плоским: все сотрудники и задачи в одном списке,
`schedule(data)` считал одно расписание, гант рисовал всех по специализациям.

Ключевое наблюдение: `occupancy` в движке — это `Map<employeeId, …>`. Движок
**не знает** про команды и знать не должен. Значит, «две доски» — это вопрос
модели данных и фильтрации вида, а не второго планировщика.

## Решение (вариант A)

Добавить команды в модель и оставить **единый глобальный план**; «два ганта» —
фильтрация вида по активной команде.

- `Team = { id, name }`, `teams: Team[]` в `AppData`; `teamId?` у `Employee` и `Task`. `SCHEMA_VERSION = 2`.
- Активная команда — вкладка вверху, **личная настройка вида** (`localStorage: resource-planner:active-team`), не в `AppData`.
- `viewData = teamView(data, activeTeamId)` (`src/engine/teams.ts`) — срез по активной команде. На нём считаются `schedule`, гант, таблица релизов, отчёт, экспорт, шаринг. Полный `data` — источник истины (`setData`, автосейв).
- Движок не тронут. Глобальные операции (сдвиг горизонта) считают `schedule(data)` по всему плану.
- Новые задачи/сотрудники стампятся `activeTeamId`.

### Подтверждённые продуктовые решения

- **Люди команд не пересекаются** (полностью раздельные составы). Поэтому срез по команде эквивалентен отдельному прогону движка — инвариант закреплён тестом `src/engine/teams.test.ts`.
- **Вид** — переключатель-вкладки в тулбаре (не два ганта стопкой).
- **Шаринг — только активная команда.** У каждой команды свой постоянный share-id и editKey: ключи `resource-planner:shareId:{teamId}` / `…:shareEditKey:{teamId}`. Публикация одной команды не затирает ссылку другой.
- **Экспорт .md / CSV — только активная команда** (то, что на экране).
- **Управление командами — в меню «Файл»:** переименовать, добавить (создаёт и переключается), удалить (каскадно уносит людей и задачи; последнюю команду удалить нельзя).

## Рассмотренные альтернативы

- **B — массив изолированных планов** (`AppData = { teams: TeamPlan[] }`): полная изоляция, но дублирует горизонт, умножает все пути `setData`/шаринга/экспорта и делает общих людей невозможными. Дороже и хуже при пересечении людей. Отклонено.
- **C — два отдельных файла/ссылки**: ноль кода, но нельзя видеть обе команды в одном приложении и общая ссылка невозможна. Отклонено (Денис хочет один переключатель).

## Последствия

- Проще: один горизонт-календарь, один файл/автосейв, отдельная ссылка на каждую команду.
- Сложнее: нужен переключатель; формы создания стампят активную команду; `assigneePool` в `TaskForm` — люди активной команды.
- Миграция v1→v2 (`src/storage/migrate.ts`) заворачивает старый план в «Команда 1»; прогоняется во всех точках загрузки (файл, автосейв, общая ссылка).

## Открытый вопрос на будущее

Нет UI «переместить сотрудника/задачу в другую команду» — при миграции все
попадают в «Команда 1», новая команда набирается заново. Для сценария Дениса
(вторая команда = новые люди) это не мешает; понадобится — добавить выбор
команды в `EmployeeManager` и `TaskForm`.
